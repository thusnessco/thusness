# Handoff request template (for Claude / design agents)

Copy everything in the **fenced block** into a new chat when you want a **new page, flow, or integration** designed *and* packaged so engineering can implement it without guesswork. Adjust the bracketed bits.

Keep this file in the repo when you refresh design zips from elsewhere — it is **not** regenerated by those exports.

---

## Paste below this line

You are producing a **design + engineering handoff** for the Thusness site (Next.js, tokens in `design_handoff_thusness/tokens.css`, brand rules in `BRAND.md`). The live codebase uses CSS variables prefixed with `--thusness-*` (see `tokens.css` in this repo).

### Goals

- **Experience:** [e.g. Orient booklet index, /readings landing, workshop signup]
- **Routes:** [list exact paths, e.g. `/foo`, `/foo/[slug]`]
- **Flexibility:** All **user-facing copy** must be editable later without redrawing layout. Prefer a **data file** + props. **Blocks** may be reordered or toggled via data or composition — do not bake copy into SVG paths or one giant non-structured HTML string for the whole page.

### Conflict rules (pick one, state it at the top of your output)

1. **Prototype wins** for spacing, type scale, and composition; `content.json` only overrides wording, OR  
2. **`content.json` wins** for wording; prototype illustrates layout only.

Default for Thusness brand: **BRAND.md + tokens win** over ad-hoc choices in the prototype (e.g. no second accent color, red dot once per page, Helvetica stack).

### Deliverables (produce all of these)

1. **Runnable reference**  
   - One self-contained artifact: **either** a single `*.html` that runs in the browser **or** a minimal React page (e.g. `prototype/YourPage.tsx` + instructions to run with Vite/Next).  
   - This file is the **visual contract** for “done” unless the checklist below says otherwise.

2. **`content/[name].json`** (or `[name].yaml`)  
   - Every **string** the user might change: titles, kickers, labels, button text, footer links, diagram labels, meta descriptions.  
   - Optional **`blocks`**: ordered array of `{ "id", "type", "variant?", "visible?" }` so sections can be reordered or hidden without code changes.  
   - **No** long prose embedded only inside React/SVG — duplicate or reference keys in this file.

3. **Presentational components**  
   - Split **layout / composition** from **data**. React (or HTML) components take a **`content` object** (typed if TS) or import the JSON.  
   - Diagrams: fixed geometry is fine; **all text** must come from `content` / JSON props.

4. **Tokens**  
   - Use **only** existing variables from `tokens.css` / `--thusness-*`. If something truly new is required, add **one** proposed token with name + value + rationale in a short `TOKEN_ADDENDUM.md` section at the bottom — do not scatter raw hex in components.

5. **`INTEGRATION.md`** (keep under one page)  
   - **Routes** → what renders where.  
   - **Data source**: static import vs `site_content` key vs Supabase table — **exact key names and TypeScript-shaped examples**.  
   - **Public vs admin** behavior if any.  
   - **TipTap / CMS** (if prose is edited in-app): allowed nodes and marks; how `content.json` maps to editor fields.  
   - **“Done when”** checklist: 5–10 bullets an implementer can tick (e.g. “no second red dot”, “footer matches BRAND”, “all strings from JSON”).

6. **Assets**  
   - List SVGs/fonts with paths. Reuse existing `assets/` where possible.

### Explicit non-goals

- Do not require a second narrative doc as the source of truth — the runnable file + JSON + INTEGRATION.md are enough.  
- Do not use Inter/Geist or icon libraries unless the request asks to break BRAND (it should not).

### Output format

- Zip or folder tree with the paths above.  
- At the root, a **`README_HANDOFF.md`** with: how to run the prototype, the conflict rule chosen, and the “done when” list repeated.
